## Deferred items

Tracked in `deferred-items.md` and below for SUMMARY visibility:

1. **Rate limiting** — pas de throttle codé. Volume pilote 11 Players × 2 jours = risque spam faible mais réel. Si abus J1, GameMaster purge via service role. v0.3: ajouter `rate_limit_window` table ou Supabase Edge Function debounce 60s/player.
2. **Realtime bell refresh** — actuellement `revalidatePath` côté Player → mentor refresh natif. Pas de WebSocket. Si J1/J2 mentor n'observe pas de refresh, il peut F5 manuel. v0.3: `supabase.channel('help_requests')` + client-side `useSubscription`.
3. **Mentor assignment scope** — pour le pilote 11 Players, tous les mentors voient toutes les demandes (cohort-wide). Pas d'assignation Player↔Mentor 1:1. v0.3: ajouter `assigned_mentor_id` sur `help_requests` ou table `mentor_assignments`.
4. **Mailto fallback** — pas de fallback `mailto:` en plus de l'in-app pour les Players qui ferment l'app. v0.3 : générer un `mailto:` draft optionnel après succès (`state.mailto`), comme `submitDeliverableFlow`.
5. **Attach mission context** — composer ne pré-remplit pas la mission courante (détectée via pathname). v0.3: ajouter `mission_context text` nullable sur table + auto-pré-remplir client-side via `usePathname`.
