Structure (commenter abondamment) :

```sql
-- RLS test suite - DATA-02
-- Pre-requis : 2 Players factices (P_A, P_B) avec leurs leader_user_id, 1 Mentor (M), 1 GM (G).
-- Renseigner les 4 UUIDs dans le bloc DECLARE ci-dessous, puis executer le fichier.

do $$
declare
  user_a uuid := '00000000-0000-0000-0000-000000000001'; -- Player A (leader)
  user_b uuid := '00000000-0000-0000-0000-000000000002'; -- Player B (leader)
  user_m uuid := '00000000-0000-0000-0000-000000000003'; -- Mentor
  user_g uuid := '00000000-0000-0000-0000-000000000004'; -- GameMaster
  player_a uuid;
  player_b uuid;
  cnt int;
begin
  select id into player_a from public.players where exists (
    select 1 from public.player_members pm where pm.player_id = players.id and pm.user_id = user_a
  ) limit 1;
  select id into player_b from public.players where exists (
    select 1 from public.player_members pm where pm.player_id = players.id and pm.user_id = user_b
  ) limit 1;
  raise notice 'Player A id=%, Player B id=%', player_a, player_b;
end $$;
```

Puis pour chaque scenario, encapsuler :

```sql
-- Scenario 1: Player A SELECT submissions
set local role authenticated;
select set_config('request.jwt.claims',
  json_build_object('sub', '00000000-0000-0000-0000-000000000001', 'role', 'authenticated')::text,
  true);

-- Doit retourner uniquement les submissions de Player A
select count(*) as visible_to_a from public.submissions;

reset role;
```

Couvrir au minimum 8 scenarios (chacun avec un commentaire `-- EXPECT:` avant) :
1. Player A SELECT submissions -> visible = ses propres rows uniquement.
2. Player A SELECT submissions WHERE player_id = player_b -> 0 rows.
3. Player A INSERT submission avec player_id = player_b -> erreur RLS (new row violates).
4. Player A INSERT submission avec player_id = player_a et submitted_by = user_a -> OK (si template existe).
5. Mentor SELECT submissions (no filter) -> count >= count(scenario 1) (Mentor voit tout).
6. Mentor INSERT pitch_scores avec juror_id = user_m -> OK.
7. Mentor INSERT pitch_scores avec juror_id = user_a -> erreur RLS.
8. GameMaster UPDATE players.score_project -> OK.
9. Player A SELECT pitch_scores WHERE player_id = player_b -> 0 rows.
10. Player A SELECT events -> >= 1 row (events authenticated_select).

Chaque assertion doit afficher `RAISE NOTICE` ou un SELECT avec label clair, pour permettre verification visuelle. Les INSERT qui doivent echouer encadres par `begin; ... rollback;` avec `exception when others then raise notice 'PASS - blocked: %', sqlerrm;`.
