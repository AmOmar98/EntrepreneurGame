Existing `computeDeliverableStatus` (lib/journey.ts) already returns `feedback_received` for V1 quand son status le devient → pas de changement requis dans journey.ts.

The Phase 1 trigger `recalc_player_score` currently sums `max(e.total_score)` over `(player, deliverable_template)` for `validated` submissions. For V1→V2 semantics ("score retenu = V2"), this is correct **iff** : on n'évalue jamais V2 puis on resoumet V1. Comme V1 est verrouillé après request_v2, seule la V2 peut devenir 'validated' après une boucle. Si V1 a été 'validated' direct (validate_v1), il n'y a pas de V2 → V1 score retenu. ✓

⇒ **Aucun changement de trigger requis**. Le brief "score retenu = V2" est déjà respecté par construction (V1 validate_v1 = pas de V2 ; V1 request_v2 → V1 status feedback_received ≠ validated → pas comptée ; V2 validate_v2 → seule la V2 est dans le agg).
