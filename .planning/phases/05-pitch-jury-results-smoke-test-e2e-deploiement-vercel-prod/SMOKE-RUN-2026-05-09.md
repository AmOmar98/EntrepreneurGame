## Findings (non bloquants pilote)

### Finding 1 : `/results` post-publish ne montre que la propre row au Player

**Severite** : MEDIUM (UX vs spec, RLS-safe)

`lib/results.ts:computeRanking` utilise `createClient()` (RLS-aware via cookies SSR), donc `players` et `pitch_scores` sont filtres par les policies. Apres publication, Player A ne voit dans le tableau classement QUE sa propre equipe (rang 1, Test Alpha) au lieu du tableau complet attendu par le smoke spec section 8.

**Comportement observe** :
- GM (`omar.ameur98`) sur `/results` : voit 4 equipes (Test Alpha 80, Test Beta 70, RLS Test A 0, RLS Test B 0), tableau preview pre-publish ET tableau publie post-publish.
- Player A (`alpha.leader@test.local`) sur `/results` post-publish : voit UNIQUEMENT Test Alpha (sa propre equipe).

**Cause** : la RLS sur `players` filtre par `player_members.user_id = auth.uid()`. La RLS sur `pitch_scores` filtre similairement. Aucune policy ne desserre l'acces public-read post-publication.

**Impact pilote** : les Players ne pourront pas voir le classement global apres le pitch final du 14 mai. Equivalent a ce que chaque equipe ne voie que son rang.

**Solutions possibles** (a discuter post-T17, pas urgent J-4) :
- A. Bypass RLS dans `computeRanking` via service-role quand `events.results_published_at IS NOT NULL` (recommande)
- B. Modifier la RLS sur `players` + `pitch_scores` pour autoriser SELECT all si event publie
- C. Vue PostgreSQL `published_results` avec security_invoker = false
